# Notes App Post-MVP Backlog

Last updated: April 1, 2026

## Purpose

This file exists to keep the MVP disciplined. Items here are valid ideas, but they are not required for Core MVP or public launch unless a later decision explicitly promotes them.

## Priority Rules

- Nothing in this file is launch-critical by default.
- No backlog feature should enter implementation until Core MVP and public-launch gates pass.
- Backlog items must be judged by whether they reinforce the capture loop, not whether they add surface area.

## Backlog Themes

### Gentle prompts

Potential value:

- reduce blank-page anxiety
- support reflective use without forcing structure

Entry criteria:

- core capture flow already feels frictionless
- prompt system does not add heavy setup or template complexity

Guardrails:

- prompts must be optional
- prompts must avoid therapeutic framing

### Reminders

Potential value:

- support routine capture and retention

Entry criteria:

- public launch already has trustworthy save/sync behavior
- reminder settings can be explained simply

### Mood markers and lightweight tags

Potential value:

- improve later recall
- give users minimal organization without building a folder system

Entry criteria:

- retrieval pain is validated in real usage
- the extra fields do not slow note completion

### Memory resurfacing

Potential value:

- make the app feel reflective and sticky

Entry criteria:

- enough stored note history exists
- search and timestamps are already reliable

Guardrails:

- resurfacing must feel supportive, not manipulative

### Native packaging or native clients

Potential value:

- stronger OS integration
- store distribution

Entry criteria:

- web limitations are proven blockers, not assumptions
- the exact missing capability is documented

## Backlog Items To Avoid By Default

- AI summary generation in MVP
- conversational AI companion behavior
- deep folder or notebook hierarchies
- collaborative sharing
- heavy personalization systems before the core loop is stable
